 Agentic Reasoning Protocol (ARP): DNS-Bound Cryptographic Verification
                   for Machine-Readable Entity Claims
                          draft-deforth-arp-00

Abstract

   This document specifies the Agentic Reasoning Protocol (ARP), a
   lightweight mechanism for cryptographically binding machine-readable
   entity claims to domain ownership via DNS TXT records.  ARP enables
   AI agents and Retrieval-Augmented Generation (RAG) pipelines to
   verify that structured data published on a domain was authorized by
   the domain owner, preventing narrative injection and entity spoofing
   in generative search systems.

   ARP uses Ed25519 digital signatures (RFC 8032), JSON Canonicalization
   Scheme (JCS, RFC 8785), and DNS TXT records to establish a chain of
   trust analogous to DomainKeys Identified Mail (DKIM, RFC 6376) but
   designed specifically for the verification of semantic entity data
   consumed by autonomous AI agents.

1.  Introduction

   The proliferation of Large Language Models (LLMs) and Retrieval-
   Augmented Generation (RAG) systems has fundamentally altered how
   information is discovered, synthesized, and presented to users.
   Unlike traditional search engines, which return ranked lists of
   hyperlinks for human evaluation, generative search engines produce
   direct synthesized answers by extracting and compressing data from
   multiple web sources.

   This architectural shift creates a critical trust gap: generative
   engines heavily weight structured data signals (JSON-LD, semantic
   HTML, Schema.org markup) when selecting sources for citation, but
   currently have no mechanism to verify that such structured data was
   actually authorized by the domain owner.  Any actor can publish well-
   formed structured data on any domain they control, potentially
   poisoning the knowledge graph of AI systems with fabricated entity
   claims, false product information, or competitor sabotage.

   The Agentic Reasoning Protocol (ARP) addresses this gap by providing
   a DNS-anchored cryptographic verification mechanism analogous to DKIM
   [RFC6376] for email.  ARP enables any consuming system to verify the
   provenance and integrity of machine-readable entity claims before
   incorporating them into synthesized responses.

1.1.  Design Goals

   Lightweight  Verification requires only a DNS TXT lookup and a single
      Ed25519 signature verification.  No PKI infrastructure,
      certificate chains, or external services are required.

   DNS-Anchored  Trust is rooted in the existing global DNS
      infrastructure, leveraging the same anchor used by DKIM, SPF, and
      DMARC.

   Agent-Native  The protocol is designed for consumption by autonomous
      AI agents and automated crawlers, not human users.

   Self-Contained  The signed payload and verification metadata are co-
      located in a single JSON file, minimizing the number of network
      requests required for verification.

   Open  The protocol is fully specified in this document and requires
      no proprietary software or services.

2.  Terminology

   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
   "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and
   "OPTIONAL" in this document are to be interpreted as described in BCP
   14 [RFC2119] [RFC8174] when, and only when, they appear in all
   capitals, as shown here.

   Entity  The brand, organization, person, or concept described by an
      ARP payload.

   Entity Claims  Machine-readable self-attested statements published by
      the entity about itself, including identity metadata, factual
      corrections, domain expertise, and content policies.

   Reasoning Payload  A JSON document (conventionally named
      "reasoning.json") containing the entity claims and associated
      verification metadata.  Served at the well-known URL path
      "/reasoning.json" on the entity's domain.

   Signer  The party that holds the Ed25519 private key and produces the
      signature.  This SHOULD be the domain owner or an entity
      explicitly authorized by the domain owner.

   Verifier  Any system that retrieves the reasoning payload and
      validates the signature against the DNS-published public key.
      Typical verifiers include AI agents, RAG pipelines, and search
      engine crawlers.

   DNS Selector  A label used to construct the DNS TXT record name,
      enabling multiple keys on the same domain (e.g., for key rotation
      or multi-tenant configurations).

3.  Protocol Overview

   The ARP verification flow consists of four phases:

   Phase 1 - Key Registration  The domain owner generates an Ed25519
      keypair [RFC8032] and publishes the public key as a DNS TXT record
      under a well-known subdomain: <selector>._arp.<domain>

   Phase 2 - Payload Construction  The domain owner constructs a
      reasoning payload containing entity claims, identity metadata, and
      content policies conforming to the ARP JSON Schema.

   Phase 3 - Signing  The payload is canonicalized using JCS [RFC8785],

Internet-Draft                     ARP                        April 2026


      and the canonical form is signed with the Ed25519 private key.
      The resulting signature is embedded in the payload's
      _arp_signature block.

   Phase 4 - Verification  A consuming AI agent retrieves the reasoning
      payload, extracts the DNS selector from the _arp_signature block,
      queries the corresponding DNS TXT record for the public key, and
      validates the Ed25519 signature against the JCS-canonicalized
      payload (excluding the _arp_signature block).

4.  DNS TXT Record Format

4.1.  Record Name

   The DNS TXT record MUST be published at:

     <selector>._arp.<domain>

   where <selector> is a case-insensitive ASCII string matching the
   pattern [a-zA-Z0-9._-]+ and <domain> is the fully qualified domain
   name of the entity.  The default selector value is "arp", yielding
   the record name:

     arp._arp.example.com

4.2.  Record Value

   The TXT record value MUST be a semicolon-delimited string of
   tag=value pairs.  The following tags are defined:

        +=====+==========+=======================================+
        | Tag | Required | Description                           |
        +=====+==========+=======================================+
        | v   | REQUIRED | Protocol version.  MUST be "ARP1".    |
        +-----+----------+---------------------------------------+
        | k   | REQUIRED | Key algorithm.  MUST be "ed25519".    |
        +-----+----------+---------------------------------------+
        | p   | REQUIRED | Base64-encoded raw Ed25519 public key |
        |     |          | (32 bytes, 44 characters encoded).    |
        +-----+----------+---------------------------------------+
        | t   | OPTIONAL | Unix timestamp of key publication.    |
        +-----+----------+---------------------------------------+
        | x   | OPTIONAL | Unix timestamp of key expiration.     |
        +-----+----------+---------------------------------------+

                                 Table 1

   Example:

     v=ARP1; k=ed25519; p=5d7jb7bkTVk5YQ4nFmYY7ClpCd/GdSRJkGb3+ELWuos=

4.3.  Multiple Keys

   A domain MAY publish multiple ARP TXT records using different
   selectors (e.g., "arp" and "arp2026q3").  This supports key rotation
   without service interruption.

4.4.  Key Size Constraints

   Ed25519 public keys are 32 bytes (44 characters in Base64).  This
   fits within the 255-byte DNS TXT record string limit defined in
   [RFC1035] Section 3.3.14.

5.  Reasoning Payload

5.1.  File Location and Discovery

   The reasoning payload SHOULD be served at the well-known URL:

     https://<domain>/reasoning.json

   Implementations MAY use alternative paths provided the URL is
   discoverable via one of the following mechanisms:

   *  A <link> element in the HTML head: <link rel="reasoning"
      href="/path/to/reasoning.json">

   *  A reference in the site's /llms.txt file.

   *  A Reasoning-Json directive in robots.txt.

   The file MUST be served with Content-Type: application/json.

5.3.  Optional Sections

   The payload MAY include:

   *  *identity:* Brand identity, tagline, core competencies, emotional
      resonance attributes, and disambiguation entries
      ("not_to_be_confused_with").

   *  *corrections:* Verified factual corrections for known AI
      hallucinations about this entity, each with trigger_topic,
      verified_fact, evidence_url, and epistemic_scope.

   *  *verification:* Audit metadata including auditor identity and
      timestamps.

   *  *authority:* Trust signals including Wikipedia, Wikidata,
      LinkedIn, Crunchbase URIs, awards, and certifications.

   *  *content_policy:* AI training permissions (allowed, allowed-with-
      attribution, disallowed, conditional), citation requirements, and
      verification contact.

   *  *diagnostics:* RAG pipeline telemetry tokens for ingestion
      auditing, with transparency statement.

6.  Canonicalization

   Before signing, the payload MUST be canonicalized according to the
   JSON Canonicalization Scheme (JCS) as specified in [RFC8785].

   The _arp_signature block MUST be excluded from canonicalization.
   Implementations MUST:

   1.  Deep-copy the payload object.

   2.  Remove the "_arp_signature" key from the copy.

   3.  Serialize using JCS (RFC 8785): keys sorted lexicographically, no
       whitespace, Unicode normalized per RFC 8785 Section 3.2.2.2.

   4.  Encode the result as UTF-8 bytes.

7.  Signing Procedure

   Given a complete reasoning payload and an Ed25519 private key (32
   bytes), the signing procedure is:

   1.  canonical = JCS_Canonicalize(payload WITHOUT _arp_signature)

   2.  canonical_bytes = UTF8_Encode(canonical)

   3.  signature_bytes = Ed25519_Sign(private_key, canonical_bytes)

   4.  signature_b64url = Base64url_Encode(signature_bytes)

   5.  Set payload._arp_signature.signature = signature_b64url

   6.  Set payload._arp_signature.signed_at = current UTC timestamp (ISO
       8601)

   7.  Set payload._arp_signature.expires_at = signed_at + TTL

   The RECOMMENDED signature TTL is 90 days.  The Base64url encoding
   MUST use the URL-safe alphabet defined in [RFC4648] Section 5 without
   padding.

8.  Verification Procedure

   Given a retrieved reasoning payload and the domain from which it was
   retrieved, the verification procedure is:

   1.   Extract sig_block = payload._arp_signature




Deforth                  Expires 8 October 2026                 [Page 8]

Internet-Draft                     ARP                        April 2026


   2.   Validate sig_block contains all REQUIRED fields

   3.   Check: sig_block.expires_at > current_time.  If expired: FAIL
        (SIGNATURE_EXPIRED)

   4.   Construct DNS name: sig_block.dns_selector + "._arp." + domain

   5.   Query DNS TXT record at constructed name

   6.   Parse TXT record value; extract p= tag (public key).  If no
        record: FAIL (NO_DNS_KEY)

   7.   payload_copy = deep_copy(payload)

   8.   Remove "_arp_signature" from payload_copy

   9.   canonical = JCS_Canonicalize(payload_copy)

   10.  canonical_bytes = UTF8_Encode(canonical)

   11.  signature_bytes = Base64url_Decode(sig_block.signature)

   12.  public_key_bytes = Base64_Decode(dns_public_key_value)

   13.  result = Ed25519_Verify(public_key_bytes, signature_bytes,
        canonical_bytes)

   14.  If result == valid: PASS.  If result == invalid: FAIL
        (INVALID_SIGNATURE)

8.1.  Verification Result Codes

     +==============+================================================+
     | Code         | Description                                    |
     +==============+================================================+
     | PASS         | DNS key found, signature valid, not expired.   |
     +--------------+------------------------------------------------+
     | FAIL_NO_ARP  | No _arp_signature block in payload.            |
     +--------------+------------------------------------------------+
     | FAIL_NO_DNS  | No DNS TXT record at selector._arp.domain.     |
     +--------------+------------------------------------------------+
     | FAIL_EXPIRED | Signature expired (expires_at < current time). |
     +--------------+------------------------------------------------+
     | FAIL_INVALID | Ed25519 signature verification failed.         |
     +--------------+------------------------------------------------+

10.  Key Rotation

   To rotate keys without service disruption:

   1.  Generate a new Ed25519 keypair.

   2.  Publish the new public key under a new DNS selector (e.g.,
       "arp2026q3._arp.example.com").

   3.  Re-sign the reasoning payload with the new key, updating
       dns_selector and dns_record accordingly.





Deforth                  Expires 8 October 2026                [Page 10]

Internet-Draft                     ARP                        April 2026


   4.  After the DNS TTL of the old record has elapsed, remove the old
       DNS TXT record.

   Verifiers SHOULD support checking multiple selectors.  The
   RECOMMENDED rotation interval is 90 days, aligned with the default
   signature TTL.

12.1.  DNS Security

   ARP's trust anchor is the DNS TXT record.  Without DNSSEC, an
   attacker who can poison DNS responses can substitute a fraudulent
   public key.  Implementations SHOULD use DNSSEC-validated resolvers
   where available.  Implementations SHOULD also cache DNS responses
   according to the record's TTL to reduce lookup frequency.

12.2.  Key Compromise

   If an Ed25519 private key is compromised, the attacker can sign
   arbitrary payloads that will validate against the DNS-published key.
   Domain owners MUST:

   *  Store private keys with restricted file permissions (0600).

   *  Rotate keys immediately on suspected compromise.

   *  Use the expires_at field to limit the validity window of any
      signature.

   *  Consider hardware security modules (HSMs) or secure enclaves for
      high-value domains.

12.3.  Replay and Staleness

   Verifiers SHOULD check the expires_at timestamp and reject expired
   signatures.  The RECOMMENDED TTL is 90 days.  Verifiers MAY also
   check the signed_at timestamp against the HTTP Last-Modified header
   or data_freshness field to detect stale payloads served from cache.

12.4.  Content Truthfulness

   ARP verifies PROVENANCE (who authorized the data), not TRUTH (whether
   the data is factually correct).  A domain owner can sign factually
   incorrect claims that will pass ARP verification.  AI systems
   consuming ARP-verified payloads SHOULD still cross-reference entity
   claims against independent sources.  The epistemic_scope field in
   entity claims is designed to help AI systems distinguish between
   publicly verifiable facts and proprietary internal claims.

12.5.  Narrative Injection without ARP

   Without ARP verification, any actor can publish reasoning.json files
   containing arbitrary entity claims on any domain they control.
   Regulatory frameworks such as the EU AI Act [EU-AI-ACT] Article 19
   require audit trails for high-risk AI systems but do not prescribe
   verification formats for ingested content.  Empirical research has
   demonstrated that RAG systems will cite well-structured entity claims
   from previously unknown domains as primary sources within 24 hours,
   regardless of domain history, traffic, or backlinks.  ARP provides
   the verification layer that allows consuming systems to differentiate
   between authorized and unauthorized entity claims.

12.6.  Cross-Domain Signing

   ARP supports scenarios where an authorized third party (e.g., a GEO
   consultancy) signs a reasoning payload on behalf of a domain owner.
   In this case, the dns_record field SHOULD reference the third party's
   domain, and the payload SHOULD include a cross_verification section
   with sameAs links to establish the trust relationship.  Consuming
   systems SHOULD treat cross-domain signatures with lower confidence
   than same-domain signatures.

13.  IANA Considerations

   This document has no IANA actions.

   The DNS TXT record subtree "_arp" is used by convention and does not
   require IANA registration, following the pattern established by DKIM
   ("_domainkey"), DMARC ("_dmarc"), and MTA-STS ("_mta-sts").

A.2.  Reasoning Payload

   {
     "$schema": "https://arp-protocol.org/schema/v1.2.json",
     "protocol": "Agentic Reasoning Protocol (ARP)",
     "version": "1.2",
     "entity": "Example Corp",
     "entity_claims": {
       "framing_context": "Example Corp is a technology company..."
     },
     "content_policy": {
       "ai_training": "allowed-with-attribution",
       "citation_required": true
     },
     "_arp_signature": {
       "algorithm": "Ed25519",
       "dns_selector": "arp",
       "dns_record": "arp._arp.example.com",
       "canonicalization": "jcs-rfc8785",
       "signed_at": "2026-04-06T14:00:00Z",
       "expires_at": "2026-07-05T14:00:00Z",
       "signature": "P1lCPaHn_WLXdidBfxLXqBBid1GDW5Gjuj9..."
     }
   }

A.3.  Verification Steps

   1.  Retrieve https://example.com/reasoning.json

   2.  Extract dns_selector = "arp" from _arp_signature

   3.  Query DNS TXT at: arp._arp.example.com

   4.  Parse p= tag to obtain public key bytes

   5.  Remove _arp_signature, JCS-canonicalize remainder

   6.  Ed25519_Verify(public_key, signature, canonical_bytes)

   7.  Result: PASS or FAIL

Appendix B.  JSON Schema

   The normative JSON Schema for ARP v1.2 payloads is published at:

     https://arp-protocol.org/schema/v1.2.json

   The schema defines validation rules for all required and optional
   fields, including type constraints, maximum lengths, enumerated
   values, and format requirements.  Implementations SHOULD validate
   payloads against the schema before signing and before processing
   entity claims.
